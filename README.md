# Provenance Guard

Provenance Guard estimates whether a piece of text-based content (a poem, a short
story excerpt, a blog post) was **written by a human or generated by an AI**. It
returns an attribution result, a confidence score, and a plain-language
**transparency label** for the reader. Creators can **appeal** a classification, and
every decision and appeal is recorded in a structured **audit log**.

Architecture, the request/appeal flow, and the full design reasoning live in
[planning.md](./planning.md). This README is the canonical record of the graded
evidence.

| Endpoint | Purpose |
|----------|---------|
| `POST /submit` | JSON body with `text` and `creator_id` (required), optional `title`. Returns attribution, confidence, transparency label, `content_id`, and per-signal breakdown. Rate-limited. |

## Detection signals (and why)

The pipeline uses **two signals that fail in different ways**. That is the whole
design idea: a single detector's blind spots become confident mistakes, but two
detectors whose weaknesses *don't overlap* can cross-check each other — and when they
disagree, the system can honestly report uncertainty instead of guessing.

### Signal A — Structural statistics (burstiness) · `signals/structural.py`

- **What it measures:** the *rhythm* of the writing — the coefficient of variation
  (std ÷ mean) of sentence lengths ("burstiness"). Deterministic and local: no
  network, same input → same score.
- **Why this signal:** human writing is *bursty* — a three-word sentence slammed
  against a forty-word one, because human thought is uneven. Autoregressive models
  optimize for the high-probability next token, which smooths output into uniform,
  medium-length sentences. Uniformity is the tell, and it's measurable with zero cost
  and full reproducibility — a good anchor against the LLM judge's non-determinism.
- **Blind spots:** short texts (too few sentences to measure variance); naturally
  uniform *human* prose (formal/technical/list writing) reads as AI; it judges form,
  never meaning, so AI told to "vary sentence length" defeats it.

> Calibration note: an earlier version also blended in type-token ratio (lexical
> diversity). Testing showed TTR contributed ≈0 across real inputs (TTR is almost
> always high) and merely diluted burstiness, so the score is now **pure
> burstiness**. TTR is still reported as evidence but not scored.

### Signal B — LLM-as-judge via Groq · `signals/llm_judge.py`

- **What it measures:** the *style/semantics* — the characteristic fingerprint of
  model output: hedged both-sides framing, empty transitional filler ("it is
  important to note", "furthermore"), over-comprehensive coverage, safe
  non-committal conclusions. Returns a probability plus a one-sentence rationale.
- **Why this signal:** statistics can't read *meaning*. A model reading holistically
  catches the register that burstiness misses. It complements Signal A precisely
  where A is weak (short or formal text).
- **Blind spots:** non-deterministic (mitigated with temperature 0, never
  eliminated); can be confidently wrong; the submitted text is a prompt-injection
  surface (guarded by wrapping it in `<CONTENT>` tags and instructing the model to
  treat it as data); and it can be biased against non-native-English / formal human
  writing. The prompt is hardened to *not* treat formality or dryness alone as AI,
  which measurably reduced that false positive in testing (a formal-human sample
  dropped from `0.8` → `0.5`).

### What I'd change for a real deployment

- **Calibrate on labeled data.** Both signals' thresholds are hand-set heuristics.
  With a labeled corpus I'd learn the fusion weights and thresholds (and fit a
  proper probability calibration — Platt/isotonic) instead of guessing them.
- **Stronger structural signal:** token-level perplexity/log-probs from a base model
  is a far better statistical detector than sentence-length variance alone.
- **Ensemble the judge:** multiple models / multiple prompts, majority vote, to blunt
  any single model's idiosyncratic blind spots and reduce variance.
- **Adversarial robustness & watermarks:** handle paraphrase attacks; incorporate
  provenance signals (C2PA, model watermarks) where available.
- **Fairness:** explicitly test and mitigate bias against non-native-English writers
  before this label ever affects a real creator.

## Confidence scoring (and why)

The system carries **two separate numbers**, and keeping them separate is the point:

- **`p_ai` ∈ [0,1]** — the fused estimate the text is AI. This is *direction*.
- **`confidence` ∈ [0,1]** — how *sure* we are of the verdict. This is *magnitude*.

**How they're computed** (`pipeline.py`, from `config.py`):

```
p_ai        = 0.5·p_ai_structural + 0.5·p_ai_llm
agreement   = 1 − |p_ai_structural − p_ai_llm|
dist        = |p_ai − 0.5| / 0.5          # 0 at coin-flip, 1 at the extremes
confidence  = dist × agreement            # BOTH must be high to be confident
```

**Why this approach:** a naive detector would just threshold one score. But two
things should lower our certainty: a fused score near the undecided middle (`dist`
low), **or** two signals that contradict each other (`agreement` low). Multiplying
them means confidence is high *only* when the evidence is both strong and internally
consistent. Disagreement between signals automatically produces low confidence →
the `uncertain` label. That's what makes a `0.51` case and a `0.95` case genuinely
different rather than a cosmetic number.

**Verdict thresholds** (applied in order): signals disagree (`|Δ| > 0.4`) →
`uncertain`; confidence `< 0.5` → `uncertain`; `p_ai ≥ 0.65` → `likely_ai`;
`p_ai ≤ 0.35` → `likely_human`; else → `uncertain`.

### Two example submissions with meaningfully different scores

Both are real outputs from testing (see `GET /log` sample below).

**High-confidence case** — uniform, generic AI prose:

| field | value |
|-------|-------|
| structural score | **1.0** |
| llm score | **0.8** |
| combined `p_ai` | **0.90** |
| **confidence** | **0.64** |
| attribution | `likely_ai` |

**Lower-confidence case** — short, neutral factual text where the signals split:

| field | value |
|-------|-------|
| structural score | **0.78** |
| llm score | **0.20** |
| combined `p_ai` | **0.49** |
| **confidence** | **0.0093** |
| attribution | `uncertain` |

The confidence differs by ~68 points (`0.64` vs `0.01`) — the signals **agreed** in
the first case and **disagreed sharply** in the second, so the score is not a
constant; it tracks real evidence strength.

### How I tested that the scores are meaningful

Beyond the two above, four deliberately chosen inputs spanning the range produced a
monotonic fused score (`0.10` human → `0.50` formal-human → `0.56` lightly-edited-AI
→ `0.76` clearly-AI), and the `fuse()` function was unit-tested against 10
hand-computed threshold cases (all passed). Investigating a mismatch (a clearly-AI
sample reading `uncertain`) drove the two calibration fixes above.

### What I'd change for a real deployment

- The confidence formula is principled but **uncalibrated** — the numbers aren't true
  probabilities. I'd fit calibration against labeled data and publish a reliability
  diagram.
- The `0.5` confidence floor is deliberately **conservative**: it under-claims AI
  (agreeing signals at `p_ai ≈ 0.75` still read `uncertain`). For a real product I'd
  tune the floor to a measured precision/recall target rather than a round number.

## Transparency label — the three variants (verbatim text)

The label communicates the result in plain language and makes the confidence level
meaningful to a non-technical reader. `{pct}` is replaced with the confidence as a
whole-number percentage (e.g. `64`). These strings are the source of truth used by
`labels.py`.

| Variant | Shown when | Exact text displayed |
|---------|-----------|----------------------|
| **High-confidence AI** | verdict `likely_ai` | `🤖 Likely AI-Generated — Our analysis indicates this content was most likely produced by an AI system (confidence: {pct}%). Two independent signals — writing-structure analysis and a language-model review — agreed on this assessment. Attribution is an estimate, not proof. If you created this yourself, you can appeal this label.` |
| **High-confidence human** | verdict `likely_human` | `✍️ Likely Human-Written — Our analysis indicates this content was most likely written by a person (confidence: {pct}%). Two independent signals agreed on this assessment. Attribution is an estimate, not a certainty — if you disagree, you can appeal this label.` |
| **Uncertain** | verdict `uncertain` (middle band, signals disagree, or low confidence) | `❓ Uncertain Attribution — We could not confidently determine whether this content was written by a human or by an AI (confidence: {pct}%). Our two signals were inconclusive or disagreed with each other. Please treat this result with caution — it should not be used as proof of authorship. You can appeal this label.` |

Quoted individually (for unambiguous copy):

- **High-confidence AI:**
  > "🤖 Likely AI-Generated — Our analysis indicates this content was most likely produced by an AI system (confidence: {pct}%). Two independent signals — writing-structure analysis and a language-model review — agreed on this assessment. Attribution is an estimate, not proof. If you created this yourself, you can appeal this label."
- **High-confidence human:**
  > "✍️ Likely Human-Written — Our analysis indicates this content was most likely written by a person (confidence: {pct}%). Two independent signals agreed on this assessment. Attribution is an estimate, not a certainty — if you disagree, you can appeal this label."
- **Uncertain:**
  > "❓ Uncertain Attribution — We could not confidently determine whether this content was written by a human or by an AI (confidence: {pct}%). Our two signals were inconclusive or disagreed with each other. Please treat this result with caution — it should not be used as proof of authorship. You can appeal this label."

## Appeals workflow

A creator who disagrees with a classification can contest it. An appeal:

1. Captures the creator's reasoning (`creator_reasoning`).
2. Logs the appeal alongside the original decision (an `appeals[]` entry on the
   decision **and** a separate `type: "appeal"` audit entry copying the original
   verdict/scores).
3. Updates the content's status to **`under_review`**.

No automated re-classification — a human reviewer decides. Authorization is
lightweight: an optional `creator_id` is enforced against the original if supplied
(mismatch → `403`); otherwise possession of the `content_id` suffices (documented
limitation, not a real auth model).

### Worked example

```bash
curl -s -X POST http://127.0.0.1:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "52a0d9cf-...", "creator_reasoning": "This is my own academic writing style.", "claimed_origin": "human"}'
```

```json
{
  "message": "appeal received",
  "content_id": "52a0d9cf-55b1-4e28-ad71-b2bf3e2f2fb6",
  "status": "under_review",
  "appeal_reasoning": "This is my own academic writing style."
}
```

## Rate limiting

`POST /submit` is rate-limited with Flask-Limiter, keyed by client IP
(`get_remote_address`), in-memory storage:

| Limit | Value |
|-------|-------|
| Per minute | **10 requests / minute / IP** |
| Per day | **100 requests / day / IP** |

Over either limit → **`429 Too Many Requests`** with a JSON body
`{"error": "rate limit exceeded", "detail": "..."}`. Only `/submit` is limited;
`/appeal`, `/log`, and `/health` are not.

### Why these numbers (defensible, not arbitrary)

- **Every `/submit` triggers a Groq LLM call** — it costs money and adds latency, so
  the endpoint is the one that actually needs protecting (this is why `/log` and
  `/appeal`, which don't call the LLM, are left unlimited).
- **10 / minute** fits the real user — a writer checking their own work. Even
  actively iterating (edit → resubmit → compare), a person submits a handful of
  times a minute; 10 leaves headroom while making a flooding script (dozens/sec)
  impossible. ~1 request / 6s sustained is plenty for a human, far too slow for abuse.
- **100 / day** caps daily cost/abuse exposure per IP — covers heavy legitimate use
  yet bounds how much a single client can spend of the LLM budget or scrape in 24h.
- **Two windows together** stop both bursts (per-minute) and slow-drip abuse that
  sneaks under a minute cap but accumulates all day (per-day).

## Known limitations

**Repetitive or minimalist poetry is likely misclassified as AI.** A villanelle, a
nursery rhyme, or any verse built on short, uniform lines and a refrain has, by
construction, very low sentence-length variation. Signal A (burstiness) measures
exactly that — coefficient of variation of sentence lengths — so it reads the
deliberate uniformity as the "machine-written" tell and scores the poem toward AI.
This is not a data problem; it's intrinsic to what burstiness measures: it equates
*uniform structure* with *AI*, which is simply false for formal verse. Worse, if the
LLM judge also reads the simple vocabulary as generic, the two signals **agree** and
the system reports a *confident* wrong answer rather than an uncertain one.

Two related weak spots, both tied to signal properties:

- **Very short submissions (< 3 sentences)** can't be scored by burstiness at all —
  variance is undefined with too few data points — so Signal A returns its
  neutral `0.5` fallback and the verdict rests on the LLM judge alone. With only one
  live signal, confidence collapses and the result is almost always `uncertain`.
- **Polished, ordinary AI prose is under-claimed.** Because of the conservative `0.5`
  confidence floor, two agreeing signals at `p_ai ≈ 0.75` still surface as
  `uncertain` rather than `likely_ai` — the system errs toward *not* accusing.

## Spec reflection

**How the spec helped.** Writing the fusion math (`confidence = dist × agreement`)
and the three label variants into `planning.md` *before* coding made M4 and M5 nearly
mechanical: `fuse()` implements the formula line-for-line, and the label builder just
maps each verdict to the pre-written verbatim string (so the README/label text match
by construction, not by luck). The "two signals that must fail in *different* ways"
principle also directly decided the second signal — an LLM judge was chosen precisely
because its blind spots (non-determinism, formal-human bias) don't overlap with
burstiness's (short/uniform text).

**How the implementation diverged.** The spec defined Signal A as burstiness **plus**
a lexical-diversity (type-token ratio) sub-score. In implementation I dropped TTR from
the score and made it *pure burstiness*, because M4 testing showed TTR contributed
≈0 on real inputs (TTR is almost always high) and only diluted the burstiness signal
and capped its maximum. TTR is still computed and reported as evidence, just not
scored. The divergence was evidence-driven, and `planning.md` was updated to record
it rather than left to drift.

## AI usage

This project was built with an AI coding assistant, directed milestone by milestone.
Two representative instances:

1. **Confidence scoring + a calibration decision.** I directed the AI to generate the
   `fuse()` scoring function from the planning spec and verify it against the
   thresholds. It produced a correct implementation (10/10 hand-computed threshold
   cases passed) — but testing revealed clearly-AI text scoring `uncertain`. The AI
   surfaced the cause and offered three fixes; I **overrode** the "lower the floor to
   make `likely_ai` reachable" option and chose to keep the conservative `0.5` floor,
   accepting that the system under-claims AI rather than risk false positives on
   formal human writing.
2. **The LLM-judge prompt.** I directed the AI to write Signal B's Groq prompt. Its
   first version false-positived on formal human prose (a monetary-policy paragraph
   scored `0.8`). I had it **revise** the prompt to explicitly not treat formality,
   dryness, or domain jargon as AI evidence; the same input then scored `0.5`,
   separating it from genuinely-AI text.
